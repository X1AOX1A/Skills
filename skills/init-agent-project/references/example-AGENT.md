# AGENT.md — 线上长输出日志挖掘协作指南

> 给在本仓库工作的 AI 助手看的入口与偏好。**每次开工前先读本文件，再读 `.agent/memory/project_progress.md`。**

---

## 1. 项目目标（一句话）

**long_log_mining** 从真实线上日志中挖掘适合长输出评测的数据，关注模型能否在较长输出过程中持续、完整、正确地执行规则，而不是单纯寻找长输入或长回答。

典型能力包括批量筛选、排序、逐项计算、保真复制与转换、固定数量生成、结构化格式、覆盖与顺序检查，以及在正确边界停止。

---

## 2. 协作风格

- **中文交流**，回答直接、有判断，不堆客套。
- **先看真实代码和数据再下结论**：涉及字段、数量、prompt、抽样、checker 或运行结果时，先核验实际文件和产物，不凭印象回答。
- **设计先行**：涉及分类边界、数据 schema、抽样口径、目录结构或大范围代码调整时，先说明方案和取舍；小而明确的修复可以直接完成并验证。
- **不过度扩展**：围绕当前任务工作，不因为某项能力容易实现就顺手扩大项目范围。
- **结论区分证据等级**：明确区分启发式候选、人工标签、程序审计事实和推断；分层抽样结果不能冒充自然分布结论。
- **临时产物统一收纳**：一次性脚本、分析结果、可视化 HTML 和原始讨论放 `.agent/workspace/<MM.DD>/`，不混入正式源码和文档。
- **保护用户数据和已有产物**：默认只读处理原始日志；运行前核对输入与输出目标，避免空结果或失败运行覆盖已有数据。

---

## 3. 任务判断原则

- 长输入、长回答都不是目标样本的充分条件。
- 优先定位最后一条真实用户任务及其必要历史，不在整段知识库或检索材料上直接做关键词判断。
- 数量必须与输出动作绑定；领域参数中的数字不能自动视为固定数量输出约束。
- 程序能够判定的事实优先交给 checker，例如数量、编号、解析、schema、长度、前后缀、禁用字符、覆盖、重复、顺序和停止位置。
- 语义忠实度、开放内容质量和复杂规则解释等无法可靠机械判定的部分，再考虑使用 judge。
- 可完全机械复核的任务与仅部分客观可核验的任务应分开建设和统计。

---

## 4. 开发与数据约定

- 可复用逻辑进入正式源码；一次性分析逻辑留在 workspace，确认稳定后再重构进入源码。
- 分类、能力体系和阈值优先数据化或配置化，避免把大量样本 ID 和易变规则写死在代码中。
- 修改任务定位、分类、checker 或 schema 时，要保留可比较的旧结果，并在固定样本上做回归核验。
- 结果应保留来源、稳定 ID、必要的回查信息和具体检查证据，不只保存一个总分或最终标签。
- 不提交凭据、内部 token、隐藏 chain-of-thought 或不必要的完整线上用户内容。
